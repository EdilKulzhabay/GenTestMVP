# Solo Kahoot: как это работает

Этот документ описывает текущую реализацию `Solo Kahoot` в проекте: кто за что отвечает, как считается время, как формируется рейтинг и как клиент с сервером взаимодействуют между собой.

Текущее состояние: `Solo` работает через `Socket.IO` (основной путь), а HTTP-endpoints остаются как fallback/совместимость.

---

## 1) Цель режима Solo Kahoot

`Solo Kahoot` — это одиночный режим с механикой Kahoot:

- один вопрос за раз;
- ограничение времени на вопрос (сейчас 15 секунд);
- очки зависят от **правильности + скорости**;
- финальный рейтинг строится по сумме очков.

Поддерживаются 2 режима:

- `daily_pack` — ежедневный пакет: **одна ranked-попытка в календарный день на пользователя**, независимо от предмета/книги; повторный `daily_pack` в тот же день отклоняется;
- `practice` — тренировочный режим, в общий рейтинг не попадает (можно проходить сколько угодно).

---

## 2) Главный принцип: server-authoritative timing

Ключевая идея — **сервер является источником истины по времени**.

Это значит:

- клиент не присылает "доверенное" время ответа;
- сервер сам фиксирует, когда начался вопрос (`questionStartedAt`);
- сервер сам вычисляет `responseTimeMs` в момент получения ответа;
- сервер сам считает очки и финальный рейтинг.

Зачем это нужно:

- защита от накрутки времени на клиенте;
- одинаковые правила для всех пользователей;
- предсказуемый и честный leaderboard.

---

## 3) Что делает сервер

### 3.1 Старт сессии

Endpoint: `POST /tests/solo/start`

Сервер:

1. Определяет `dailyPackId` (детерминированно по дате и параметрам пакета: предмет, книга и т.д.).
2. Для `daily_pack`: если у пользователя уже есть **любая** `ranked`-попытка с `createdAt` с начала текущего календарного дня — возвращает ошибку (лимит «один daily pack в день» глобально).
3. Определяет тип попытки:
   - `ranked`, если режим `daily_pack` и лимит не исчерпан;
   - иначе `practice`.
4. Создаёт `SoloSession`:
   - `soloSessionId`
   - `currentQuestionIndex = 0`
   - `questionStartedAt = now (server time)`
   - `answers = []`
5. Возвращает тест и мета-информацию сессии.

### 3.2 Realtime gateway (Socket.IO)

Сервер поднимает WS-gateway и авторизует сокет по JWT:

- токен берётся из cookie `token` или из `handshake.auth.token`;
- после верификации сокет привязывается к `userId`;
- для Solo используются события:
  - `solo:join`
  - `solo:answer`
  - `solo:finish`

### 3.3 Приём ответа на вопрос

Основной путь: `solo:answer` (Socket.IO event)  
Fallback: `POST /tests/solo/answer`

Вход:

- `soloSessionId`
- `questionIndex`
- `selectedOption`

Сервер:

1. Проверяет, что сессия существует, принадлежит пользователю и не завершена.
2. Проверяет, что `questionIndex` совпадает с `currentQuestionIndex`.
3. Считает серверное время ответа:
   - `elapsedMs = now - questionStartedAt`
   - `responseTimeMs = min(elapsedMs, questionTimeLimitMs)`
4. Проверяет правильность ответа.
5. Считает очки за вопрос.
6. Сохраняет запись ответа в `SoloSession.answers`.
7. Переводит сессию на следующий вопрос:
   - увеличивает `currentQuestionIndex`
   - обновляет `questionStartedAt = now`
8. Если вопрос был последним — помечает сессию `isFinished = true`.

### 3.4 Завершение сессии и запись результата

Основной путь: `solo:finish` (Socket.IO event)  
Fallback: `POST /tests/solo/finish`

Сервер:

1. Проверяет, что ответы есть для всех вопросов.
2. Суммирует `questionScore` -> `finalScore`.
3. Считает `correctCount`.
4. Создаёт `SoloAttempt` (история/рейтинг):
   - `dailyPackId`
   - `attemptType` (`ranked/practice`)
   - `finalScore`
   - агрегаты по попытке.
5. Для `ranked` считает место (`rank`) среди ranked попыток.
6. Возвращает финальный результат клиенту.

---

## 4) Что делает клиент

### 4.1 На старте теста

Клиент вызывает `POST /tests/solo/start` и получает:

- вопросы;
- `soloSessionId`;
- `soloQuestionStartedAt`;
- лимит времени на вопрос.

### 4.2 Во время прохождения

Клиент отвечает за UX:

- показывает **один вопрос**;
- рисует локальный визуальный таймер;
- при старте подключается к сокету и делает `solo:join`;
- при клике "дальше" (или по авто-логике) отправляет ответ через `solo:answer`.

Важно: клиент не считает финальные очки и не определяет официальное время ответа.

### 4.3 На финише

После последнего вопроса клиент вызывает `solo:finish` и показывает:

- итоговые правильные ответы;
- `finalScore`;
- тип попытки (`ranked/practice`);
- место в рейтинге (если ranked).

---

## 5) Формула очков

Для каждого вопроса:

- если ответ неверный: `0`
- если верный:

`Score = 1000 * (0.3 + 0.7 * (remaining / 15))`

где:

- `remaining` — оставшиеся секунды до конца 15-секундного окна;
- `remaining` не может быть меньше 0.

Следствия:

- быстрый правильный ответ даёт больше очков;
- медленный правильный ответ даёт меньше, но > 0;
- неверный/просроченный — 0.

---

## 6) Как строится рейтинг

Источник рейтинга — коллекция `SoloAttempt`:

- в leaderboard попадают только `ranked`-попытки;
- сортировка: `finalScore DESC`;
- при равных очках выше тот, кто зафиксирован раньше (tie-break по времени создания попытки).

`practice` попытки:

- показывают очки пользователю;
- в официальный рейтинг не включаются.

---

## 7) Распределение ответственности

### Сервер (доменная логика, источник истины)

- authoritative timing;
- валидация порядка вопросов;
- проверка правильности ответов;
- расчёт очков;
- финализация попытки и запись в рейтинг;
- выдача rank/top.

### Клиент (UI/UX и оркестрация экрана)

- показывает вопрос, прогресс, таймер;
- собирает пользовательский ввод;
- держит realtime-соединение по `Socket.IO`;
- отправляет `join -> answer -> ... -> finish` в правильной последовательности;
- рендерит финальный экран и лидерборд.

---

## 8) Ключевые модели данных

### `SoloSession`

Техническая сессия прохождения:

- активный вопрос;
- серверный timestamp старта вопроса;
- по-вопросные ответы и времена;
- статус завершения.

### `SoloAttempt`

Итог попытки для статистики/рейтинга:

- `dailyPackId`;
- `attemptType`;
- `finalScore`;
- агрегированные метрики;
- timestamp для tie-break.

---

## 9) Почему такое разделение правильно

Плюсы:

- честный рейтинг;
- защита от клиентских манипуляций;
- простая масштабируемая модель (session -> attempt);
- прозрачный audit trail по каждому вопросу.

Риски/ограничения:

- чувствительность к задержкам сети (responseTime измеряется на сервере, это ожидаемо и честно для всех);
- нужен аккуратный UX при сетевых лагах (retry, состояние отправки, блокировка двойной отправки).

---

## 10) Краткий sequence flow

1. Клиент: `POST /tests/solo/start`
2. Сервер: создаёт `SoloSession`, отдаёт `soloSessionId`, `questionStartedAt`
3. Клиент: открывает Socket.IO и отправляет `solo:join`
4. Сервер: подтверждает актуный `currentQuestionIndex` и `questionStartedAt`
5. Клиент: показывает вопрос N
6. Клиент: `solo:answer (N, selectedOption)` через сокет
7. Сервер: считает `responseTimeMs`, `questionScore`, двигает сессию на N+1
8. Повтор шагов 5-7 до конца
9. Клиент: `solo:finish` через сокет
10. Сервер: создаёт `SoloAttempt`, считает rank, отдаёт итог
11. Клиент: показывает результат и рейтинг

---

## 11) Транспортный контракт (актуально)

### HTTP

- `POST /tests/solo/start` — создать сессию и получить мета-данные.
- `GET /tests/solo/leaderboard` — получить таблицу лидеров.
- `POST /tests/solo/answer`, `POST /tests/solo/finish` — fallback путь.

### Socket.IO

- `solo:join` — вход в Solo-сессию и синхронизация состояния.
- `solo:answer` — ответ на текущий вопрос (сервер считает время и очки).
- `solo:finish` — завершение сессии и финальный результат.
