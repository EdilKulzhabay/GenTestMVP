# 1. Product Vision & MVP Boundaries

**Цель:** Зафиксировать общую концепцию продукта и явные границы MVP.

### Product Vision

**Цель продукта:**

Создать образовательную платформу, которая помогает школьникам **системно и эффективно готовиться к ЕНТ**, используя:

- Структурированный контент школьных книг
- Автоматическую генерацию тестов
- Разбор ошибок с привязкой к источникам
- Персональный учебный путь (roadmap)

**Основная гипотеза MVP:**

Пользователи, уже мотивированные на подготовку к ЕНТ, будут регулярно использовать продукт, если он **быстро приводит к пониманию слабых мест** и **даёт чёткий следующий шаг**, даже без геймификации.

**Core Loop продукта:**

```
Открыть роадмап
→ Пройти короткий тест
→ Получить разбор ошибок
→ Обновить персональный путь
→ Перейти к следующему узлу
```

**Целевая аудитория:**

- Ученики 9–11 классов
- Уже готовятся к ЕНТ
- Используют мобильные устройства как основной девайс

### MVP Boundaries

*Что входит в MVP и что остается за его пределами*

**✅ Входит в MVP:**

- Авторизация и профиль пользователя
- Canonical roadmap по предмету
- Personal roadmap, формируемый после первого теста
- Express-тесты (5–10 вопросов)
- Асинхронная генерация тестов
- Прохождение теста
- Результаты и разбор ошибок
- Рекомендации "что изучать дальше"
- Markdown-контент книг + индекс
- Preset / Prompt table для генерации

**❌ НЕ входит в MVP:**

- UI-геймификация (XP, лиги, лидерборды)
- Социальные функции
- Публичные лендинги и SEO
- Монетизация
- Адаптивное обучение на ML
- Автоматический парсинг PDF

<aside>
⚠️

Данные для геймификации (progress delta, reward hooks) **считаются**, но **не отображаются в UI**

</aside>

# 3. User Flows & Business Flows

**Цель:** UX-архитектор рисует flow diagram без UI.

### Business Flow

**Основной бизнес-процесс:**

1. Пользователь выбирает предмет для подготовки
2. Система предоставляет canonical roadmap (структура знаний)
3. Пользователь выбирает узел для изучения
4. Система генерирует тест асинхронно (5–10 вопросов)
5. Пользователь проходит тест
6. Система анализирует ответы и формирует:
    - Результаты (score %)
    - Разбор ошибок с источниками
    - Рекомендации по дальнейшему обучению
7. Система обновляет personal roadmap пользователя
8. Цикл повторяется

### User Flow

**Flow A — Первый вход → Первый тест:**

```
Login
→ Select Subject
→ Canonical Roadmap
→ Click available node
→ Generate Test Job
→ Job Status (queued / generating)
→ Start Test Session
```

**Flow B — Тест → Результат → Personal Roadmap:**

```
Test Session
→ Submit answers
→ Result + error analysis
→ Recommendations
→ System: update progress
→ Personal roadmap becomes available
```

**Flow C — Возвращающийся пользователь / Resume:**

```
Login
→ Continue learning
→ Personal Roadmap
→ Next recommended node
→ Generate Test
```

**Цель:** Архитектура модулей и границы ответственностей стримов.

### Модули и стримы

**Основные модули системы:**

**1. Auth / Profile**

- Авторизация пользователей (телефон + OTP, OAuth)
- Регистрация с постепенным заполнением профиля
- Управление профилями
- Гостевой режим для прохождения тестов

**2. Roadmap Service**

- Canonical roadmap (статичный JSON)
- Структура знаний по предметам

**3. Progress Service**

- Personal roadmap (проекция canonical + метаданные)
- Отслеживание прогресса пользователя

**4. Content / Books**

- Markdown-контент книг
- Индексация и поиск
- Структура хранения знаний

**5. Preset / Prompt Service**

- Управление preset-конфигурациями
- Шаблоны промптов для LLM
- Оптимизация токенов

**6. Test Generation (Async Jobs)**

- Асинхронная генерация тестов
- Управление очередью задач
- Статусы: queued, generating, ready, failed, canceled

**7. Test Sessions**

- Прохождение тестов
- Сохранение ответов
- Возврат к незавершённым сессиям

**8. Analysis & Recommendations**

- Анализ ошибок
- Формирование рекомендаций
- Обновление personal roadmap

---

**Canonical vs Personal Roadmap:**

- **Canonical** — статичный JSON со структурой знаний
- **Personal** — проекция canonical + пользовательские метаданные

```jsx
PersonalNodeProgress {
  nodeId
  status
  completionPercentage
  averageScore
}
```

# 5. API Contracts

**Цель:** Определить, что будет ожидать фронтенд и какие состояния нужно продумать.

### Контракты

**API Endpoints:**

**Генерация тестов:**

```
POST /api/tests/generate → { jobId }
GET  /api/tests/jobs/:jobId → { status, testId? }
GET  /api/tests/:testId → { test }
```

**Статусы генерации:**

- `queued` — в очереди
- `generating` — генерируется
- `ready` — готов
- `failed` — ошибка
- `canceled` — отменён

<aside>
⚠️

UI **обязан** обрабатывать все состояния генерации

</aside>

**Test Session:**

```jsx
// Сохранение ответа
POST /api/sessions/:sessionId/answers
{
  questionId: string
  answer: string
}

// Завершение теста
POST /api/sessions/:sessionId/submit
→ {
  score: number // %
  errors: Array<{
    questionId: string

    source: {
      book: string
      chapter: string
      page: number
    }
```

```jsx
PromptTemplate {
```

### UX Flow

**Ключевые сценарии:**

1. **Первое использование**
    - Минимум решений до первого теста
    - Быстрый старт с canonical roadmap
    - Явная индикация процесса генерации
2. **Прохождение теста**
    - Одно действие на экран
    - Возможность вернуться к вопросам
    - Сохранение прогресса
3. **Анализ результатов**
    - Чёткий score (%)
    - Разбор каждой ошибки
    - Ссылки на источники (книга/параграф/страница)
4. **Продолжение обучения**
    - Быстрый доступ к personal roadmap
    - Рекомендации следующего шага
    - Resume незавершённых тестов

### Упрощения и оптимизации

**UX-принципы для снижения когнитивной нагрузки:**

✅ **Mobile-first**

- Весь UX проектируется под мобильные устройства

✅ **1 основное действие на экран**

- Убираем множественный выбор
- Фокус на следующем шаге

✅ **Progressive disclosure**

- Детали по запросу
- Не перегружаем экраны информацией

✅ **Явные состояния**

- Ожидание (queued, generating)
- Ошибки (failed)
- Успех (ready)

✅ **Минимизация шагов до ценности**

- От входа до первого теста: 3 клика
- От результата до следующего действия: 1 клик

**Что убрали из MVP:**

- Геймификация в UI (считается в бэкенде)
- Социальные функции
- Сложные настройки тестов

# 8. Auth & Registration Flow

**Цель:** Детально описать процесс авторизации и регистрации с учетом гостевого режима и монетизации.

<aside>
📱

**Mobile-first:** Все флоу проектируются под мобильные устройства с адаптацией для десктопа

</aside>

### Концепция авторизации

**Основная идея:**

- Пользователь может **начать использовать продукт без регистрации**
- Доступ к **ценным функциям** (разбор ошибок, personal roadmap) требует авторизации
- Это создаёт **естественную точку конверсии** и закладывает основу для **монетизации**

**Граница авторизации (Paywall Gate):**

| Функция | Без авторизации | С авторизацией | Будущая монетизация |
| --- | --- | --- | --- |
| Просмотр canonical roadmap | ✅ Доступно | ✅ Доступно | Бесплатно |
| Прохождение тестов | ✅ Доступно | ✅ Доступно | Бесплатно |
| Базовый результат (score %) | ✅ Доступно | ✅ Доступно | Бесплатно |
| Разбор ошибок с источниками | ❌ Требует auth | ✅ Доступно | 🔐 Premium |
| Personal roadmap | ❌ Требует auth | ✅ Доступно | 🔐 Premium |
| Рекомендации следующих шагов | ❌ Требует auth | ✅ Доступно | 🔐 Premium |
| История прохождения тестов | ❌ Требует auth | ✅ Доступно | Бесплатно |

### Методы авторизации

**1. Телефон + OTP (Основной метод)**

- Ввод номера телефона
- Отправка OTP-кода через SMS
- Верификация кода
- Автоматическая регистрация при первом входе

**2. OAuth провайдеры**

- **Google** — основной OAuth-провайдер
- **EduKey** — интеграция с сервисами бизнес-владельца
- Возможность добавления других провайдеров

**3. Гостевой режим (Guest)**

- Пользователь без аккаунта
- Временная сессия в localStorage/cookies
- Ограниченный доступ к функциям

### User Flow: Первый визит (Guest)

**Сценарий A: Пользователь не хочет регистрироваться сразу**

```jsx
Открыть приложение (Guest Mode)
→ Выбрать предмет
→ Просмотреть canonical roadmap
→ Выбрать узел
→ Сгенерировать и пройти тест
→ Получить базовый результат (score %)
→ 🚪 GATE: "Хотите увидеть разбор ошибок?"
   → [Зарегистрироваться] → Flow регистрации
   → [Позже] → Остаться в Guest Mode
```

**Retention Hook в Guest Mode:**

- После завершения теста показываем **тизер результатов**:
    - "Вы ответили правильно на 7 из 10 вопросов"
    - "Хотите узнать, какие темы нужно подтянуть?"
    - Размытый preview разбора ошибок
    - CTA: "Зарегистрироваться и получить разбор"

### User Flow: Регистрация через телефон

**Flow B: Регистрация нового пользователя**

```jsx
Нажать "Зарегистрироваться"
→ Экран: Выбор метода регистрации
   - [По номеру телефона] ← основной метод
   - [Через Google]
   - [Через EduKey]
→ Ввести номер телефона
   - Формат: +7 (___) ___-__-__

→ ✅ Успешная регистрация
→ Экран: "Расскажите о себе" (Onboarding)
   - Имя (опционально)
   - Класс (9 / 10 / 11) — важно для контента
   - [Пропустить] — возможность заполнить позже
```

### Постепенное заполнение профиля

**Концепция Progressive Profiling:**

Не требуем всю информацию сразу — собираем данные по мере использования продукта.

**Минимальные данные при регистрации:**

- Телефон (для Phone + OTP) ИЛИ Email (для OAuth)
- Класс (опционально, но важно)

**Дополнительные поля (заполняются позже):**

- Имя
- Фамилия
- Школа
- Город
- Предметы для подготовки
- Целевой балл ЕНТ

**Когда просить дозаполнить:**

1. **Мягкие напоминания** в UI:
    - "Заполните профиль, чтобы получать персональные рекомендации"
    - Показываем после 2-3 тестов
2. **Естественные моменты**:
    - Перед стартом нового предмета: "Какой у вас класс?"
    - При первом открытии personal roadmap: "Укажите целевой балл"
3. **Не блокируем** основной функционал

### API Contracts: Auth

**Phone + OTP Registration/Login:**

```jsx
// Шаг 1: Запрос OTP
POST /api/auth/phone/request-otp
{
  phone: "+77001234567"
}
→ Response:
{
  success: true,
  otpSent: true,
  expiresIn: 60, // seconds
  requestId: "req_abc123" // для tracking
}

// Шаг 2: Верификация OTP
POST /api/auth/phone/verify-otp
{
  phone: "+77001234567",
  otpCode: "123456"
}
→ Response (новый пользователь):
{
  success: true,
  isNewUser: true,
  accessToken: "jwt_token",
  refreshToken: "refresh_token",
  user: {
    id: "user_123",
    phone: "+77001234567",
    profileComplete: false // нужно заполнить профиль
  }
}

→ Response (существующий пользователь):
{
  success: true,
  isNewUser: false,
  accessToken: "jwt_token",
  refreshToken: "refresh_token",
```

### UX States & Edge Cases

**Состояния авторизации в UI:**

1. **Guest** — без аккаунта
    - Показываем баннер "Зарегистрируйтесь для полного доступа"
    - CTA на ключевых экранах
2. **Authenticated** — залогинен
    - Полный доступ к функциям
    - Персональные рекомендации
3. **Authenticated + Incomplete Profile** — профиль не заполнен
    - Мягкие подсказки о заполнении
    - Не блокируем функционал

**Edge Cases:**

❌ **OTP не приходит**

- Показываем "Не пришел код?"
- Кнопка "Отправить повторно" (через 60 сек)
- Альтернатива: "Попробовать другой способ входа"

❌ **Неверный OTP-код**

- Показываем ошибку "Неверный код"
- 3 попытки, затем блокировка на 5 минут
- Предложение отправить код повторно

❌ **OAuth провайдер недоступен**

- "Сервис временно недоступен"
- Предложение использовать телефон + OTP

❌ **Пользователь зарегистрирован через телефон, пытается войти через Google с тем же email**

- Backend: Связать аккаунты (account linking)
- UI: "Этот email уже используется. Связать аккаунты?"

❌ **Потеря сессии во время прохождения теста**

- Сохраняем прогресс в localStorage
- После авторизации восстанавливаем сессию

### Mobile-First Design Considerations

**Ключевые принципы для мобильных устройств:**

✅ **Большие кнопки и области ввода**

- Минимум 44×44pt для touch targets
- Крупный шрифт для OTP-кода

✅ **Автозаполнение OTP**

- iOS: автоматическое чтение SMS
- Android: SMS Retriever API

✅ **Минимум ввода текста**

- Приоритет: кнопки, селекторы
- Клавиатура с правильным inputMode:
    - `tel` для телефона
    - `numeric` для OTP

✅ **Adaptive для планшетов и десктопа**

- Мобильные экраны: 1 колонка
- Планшеты: 2 колонки для форм
- Десктоп: центрированные формы с max-width

✅ **Быстрая навигация**

- Свайпы для возврата назад
- Нативные паттерны iOS/Android

### Security & Privacy

**Безопасность:**

- OTP коды истекают через 5 минут
- Rate limiting на отправку OTP (max 3 в час)
- JWT токены с коротким TTL (15 мин access, 7 дней refresh)
- HTTPS для всех запросов

**Приватность:**

- Минимальный сбор данных
- Телефон используется только для авторизации
- Опциональные поля остаются опциональными
- Согласие на обработку персональных данных при регистрации
